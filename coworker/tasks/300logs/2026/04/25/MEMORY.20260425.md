# MEMORY.20260425.md
## Daily Memory - 2026-04-25

- Fixed the `browser4-cli` Rust dead-code warnings reported by `cargo run open` by changing `sdks/browser4-cli/src/lib.rs` to export `state` as a public library module instead of keeping it as a private lib-only module. That preserved the existing binary behavior while making the bin-used state helpers part of the library surface so Cargo no longer warns when compiling the separate `browser4-cli` lib target.
- Confirmed the result with `cargo check` and `cargo test --bin browser4-cli --quiet` in `sdks/browser4-cli`.
- Key lesson: when a Cargo package builds both a binary and a library, helpers used only by the binary can still trigger dead-code warnings if they live in a private library module; either keep them out of the library target or export them intentionally so the crate boundary matches actual usage.
- Fixed `sdks/browser4-cli/src/daemon.rs` startup-log handling so the CLI now creates Browser4 startup logs under the per-user temp root `temp/browser4-$USER/tmp/cli`, writes explicit daemon-side startup progress and failure lines into the same log file, and surfaces log-initialization/write failures on stderr instead of silently losing them. The daemon tests were updated to match the current `ROOT.md`-based repo-root detection already present in the worktree, and `cargo check --quiet` plus `cargo test --bin browser4-cli daemon --quiet` passed in `sdks/browser4-cli`.
- Key lesson: startup diagnostics should live in a predictable temp location tied to the effective user, and the launcher should append its own progress/error events to the same file as child stdout/stderr so spawn-time failures are still captured even when the server process never becomes ready.
