# MEMORY.20260415.md
## Daily Memory - 2026-04-15

- Reworked `browser4-cli batch` from a recursive frontend-only wrapper into a real backend-executed flow: the Rust CLI now parses batch input, compiles each logical CLI command into backend batch steps, preserves local session/selector/mouse state around the response, and saves snapshot/screenshot artifacts after a single `command_batch` MCP call. On the server side, `pulsar-rest` now exposes `command_batch`, executes supported browser/session commands sequentially in one request, and returns structured per-step results so the CLI can keep existing batch UX such as `--json`, `--bail`, explicit snapshot output, and ordered failure reporting. Lesson: when batching an existing CLI surface for performance, keep the CLI as the syntax/state adapter but move execution to one backend request, and model multi-step commands like `open <url>` as one logical CLI command backed by multiple internal batch steps so behavior stays correct without reintroducing per-command HTTP overhead.
- Updated `sdks/browser4-cli` local daemon startup so `ensure_server_running()` now prefers launching Browser4 from a detected repo root via Maven `spring-boot:run` targeting `pulsar-rest`, while retaining the existing `Browser4.jar` path as a fallback when no checkout is available. Added focused daemon unit tests for Browser4-root detection and Maven launch-spec construction, updated CLI docs to describe the new localhost startup preference, and preserved managed-process tracking by recording the launcher target plus resolving the child JVM PID on Windows. Lesson: when a CLI should use the matching in-repo server by default, make the preferred startup path explicit but keep the packaged runtime as a fallback so installed binaries do not regress; also avoid whole-crate format runs in a dirty worktree because they can create unrelated diffs. Validation note: focused daemon tests and root Maven compile passed; the long `browser4-cli` browser-backed E2E suite still reproduces an existing local transport failure during `test_e2e_interaction_commands` (`goto` lost the MCP server connection), so it was not a clean end-to-end signal for this task.
