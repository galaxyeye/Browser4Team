# MEMORY.20260414.md
## Daily Memory - 2026-04-14

- Added a Unix `sdks/browser4-cli/install.sh` installer that detects supported package managers, installs missing Java 17+, Google Chrome, Rust, and native build dependencies, downloads the latest released `Browser4.jar`, and installs `browser4-cli` from the latest tagged Browser4 source into `~/.local/bin`. Updated the CLI and root README docs to advertise the installer and the Windows release-asset fallback. Lesson: when a product ships only part of its release surface as binaries, the installer should pin all downloaded artifacts to the same release tag and make the source-build fallback explicit so users still get a consistent end-to-end install experience.

- Removed the obsolete `sdks/browser4-cli/install.ps1` installer, deleted all remaining code/docs references to it, and rewrote the Windows CLI installation guidance to use the latest released `Browser4.jar` plus manual `cargo install --path . --locked` setup. Also updated the Unix installer error message so Windows users are redirected to the supported manual path instead of a deleted script. Lesson: when retiring an install surface, remove the artifact and every discoverability path together, and leave Windows users with a concrete supported fallback instead of a dead link.

- Refactored `sdks/browser4-cli/tests/e2e.rs` into smaller scenario-focused Rust E2E cases, kept the custom sequential harness, added per-test duration reporting, and tuned Browser4 restart boundaries so the split suite stays stable while still surfacing slow cases like interaction and tab coverage. While validating, rebuilt an invalid `browser4/browser4-agents/target/Browser4.jar` artifact before rerunning the suite, which finished with 10 passing E2E tests and printed timings for each case. Lesson: when breaking a long end-to-end harness into finer cases, preserve the previously stable state boundaries (especially for low-level keyboard/mouse flows) and report timings directly from the harness so new bottlenecks are obvious without sacrificing determinism.

- Reordered `.github/workflows/ci.yml` so the Docker-backed `browser4-cli` E2E stage now runs immediately after the application health check, and the reusable Maven `run-tests` stage executes only after the CLI check passes. This keeps the existing build/startup flow intact while short-circuiting the broader test workload when the CLI surface is already broken. Lesson: when one end-to-end suite is a leading indicator for the rest of the pipeline, run it before the wider test matrix so CI fails faster and spends less time on likely-follow-on failures.

- Implemented `browser4-cli batch` so multiple CLI commands can run in one process via quoted command arguments or `--json` stdin arrays, with optional `--bail` fail-fast behavior and inheritance of the top-level `--server` / `-s` flags across nested commands. Added batch parsing coverage in `src/args.rs`, help/command catalog updates, README usage examples, and a browser-backed E2E scenario that verifies argument mode, JSON stdin mode, non-bailing continuation, and fail-fast stopping. Lesson: when adding a meta-command over an existing CLI, reuse the normal dispatch path instead of creating a parallel executor so state persistence, startup logic, alias rewriting, and post-command behavior stay consistent across single-command and batched flows.
